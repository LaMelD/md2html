## 1. Linear 이슈 등록 방식 개선 논의 (0:19~15:00)

### 인라인 vs 파일 분리 방식 비교

이상준 매니저가 전날에 이어 Linear 이슈 등록 테스트 결과를 공유했다. 테스트는 두 가지 방식으로 진행되었다. 첫 번째(v1)는 플랜 내용을 서브 에이전트에게 인라인으로 직접 주입하는 방식이고, 두 번째(v2)는 작업 상세 내용을 별도 파일로 저장하고 필요한 경우 플랜의 특정 줄을 참조하도록 명시하는 방식이다. 비교 평가 결과 v2 방식의 완성도가 더 높은 것으로 나타났으며, 환각도 거의 없었다. 다만 메인 에이전트의 토큰 사용량이 v2에서 다소 증가하였다.

박범균 소장은 이슈 단위로 파일을 생성하고 쪼개는 방식에 의문을 제기했다. 대안으로 플랜 작성 시 처음부터 작업 단위별 체크리스트를 하단에 포함하고, 해당 체크리스트에서 플랜의 참조 줄 번호를 직접 명시하는 방식을 제안했다. 이렇게 하면 별도 파일 생성 없이 플랜 한 문서만 메인 에이전트가 로드하면 되고, 서브 에이전트는 해당 체크리스트의 관련 부분만 확인하여 작업할 수 있다는 설명이었다. Linear에는 마일스톤 하나에 이슈 하나 정도만 간략히 등록하고, 상세 작업 추적은 플랜 내 체크리스트로 대체하자는 결론을 도출했다.

### Linear 이슈 추적의 핵심 가치

박범균 소장은 Linear를 사용하는 핵심 이유가 "작업 중 발견된 이슈를 실시간으로 등록하고 나중에 추적하기 위함"임을 강조했다. 계획적으로 예측 가능한 작업은 간략히 등록해도 되지만, 작업 도중 발생하는 예상치 못한 이슈나 워크트리 분기가 필요한 상황은 반드시 이슈로 등록해야 한다고 설명했다. 이슈를 등록하지 않으면 나중에 추적이 불가능해지므로 이 기능 자체를 축소할 수는 없다고 못 박았다.

이상준 매니저는 서브 에이전트 동작 중 작업 실패나 우회가 발생했을 때 Linear 이슈에 코멘트로 기록하는 방식을 적용하고 있다고 공유했다. 다만 현재 구성상 레지스터 단계와 익스큐트 단계가 분리되어 있어 메인 에이전트가 실시간으로 감지하기 어려운 구조임을 설명했다.

## 2. PHP 단위 테스트 방법론 논의 (7:45~16:00)

### Python 회귀 테스트 구조 소개

박범균 소장은 Python 기반 프로젝트에서 운영 중인 단위 테스트 구조를 소개했다. 현재 테스트 항목이 700개 이상이며 전체 회귀 테스트 실행 시 약 5분이 소요된다. 테스트 코드는 삭제하지 않고 누적 보관하며, 새로운 코드 작성 시에는 해당 기능의 테스트만 수행하고, 기존 코드를 수정할 때는 전체 700개 테스트를 실행하여 사이드 이펙트를 검증하는 회귀 테스트 방식으로 운영한다. 이 방식으로 코드 작업 후 오류 발생 빈도가 크게 줄었으며, AI가 작업을 완료했는지 여부를 객관적으로 확인할 수 있다고 설명했다.

### PHP 단위 테스트의 어려움

PHP는 웹 서버 환경에 종속되어 있어 Python처럼 파일을 임포트하고 함수를 직접 실행하는 방식의 단위 테스트가 구조적으로 어렵다는 문제가 제기되었다. 주정민 매니저는 현재 PHP 테스트 시 기댓값 검증 로직을 위해 함수 호출마다 파일을 별도 생성하고 삭제하는 과정이 반복되어 토큰 소모가 크다고 설명했다. 또한 PHP 뷰(View) 테스트는 실제 브라우저를 통한 확인이 필요해 별도 방법론이 요구된다고 덧붙였다. 김태규 매니저도 테스트로 인한 토큰 소모가 과도하다는 점에 동의했다.

PHP 클래스 임포트 후 직접 호출하는 방식이 이론적으로는 가능하지만, 기본 로드 경로 의존성이나 컨트롤러를 통한 호출 제약 때문에 예외 상황이 많이 발생할 것이라는 우려가 나왔다. 박범균 소장은 단위 테스트 자체는 고성능 모델(Opus 등)을 쓸 필요가 없으며 결과 확인만 하면 되지만, 기댓값 정의가 플랜 작성 시 포함되면서 Opus 토큰 사용량이 간접적으로 늘어난다는 구조적 문제를 지적했다. PHP 프로젝트에 맞는 단위 테스트 방법론은 별도 정립이 필요하다는 결론으로 마무리되었다.

## 3. 세션 사용량 관리 및 운영 전략 (15:36~22:00)

### 세션 초기화 시점 전략

서진우 매니저는 오전 8시에 스크립트로 세션을 자동 시작하는 방식을 사용 중이라고 공유했다. 이에 박범균 소장은 12시 초기화 + 오후 작업 진입 시점을 고려하면 오전 7시 초기화가 가장 효율적일 것이라고 제안했다. 김경승 팀장은 12시 초기화 시 오후 5시 30분 경에 세션이 소진되어 퇴근 직전 집중 작업이 불가능해진다는 현실적 문제를 제기했다. 박범균 소장은 오후 4시 정례 회의가 보통 5시까지 이어지는 점을 감안하면 실질적 영향이 없다고 반론했다.

이상준 매니저는 하루 두 사이클 기준 5~6일치 사용량이며, 세 사이클 이상 사용 시 3~4일 만에 소진될 것이라고 러프한 계산 결과를 공유했다. 박범균 소장은 사용량 부족이 현실적 문제임을 인정하면서도, 팀원 개인별 200달러 플랜 추가 결제는 비용 대비 관리 효율 측면에서 선뜻 결정하기 어렵다고 언급했다.

### Codex 테스트 현황

## 4. 토큰 사용량 증가 원인 분석 (22:00~54:00)

### Linear 등록 시 아웃풋 토큰 과다 소모

박범균 소장은 코드 작업은 수만 줄을 작성해도 아웃풋 토큰 비중이 크지 않은 반면, Linear 이슈 등록은 순수 자연어 텍스트(한글) 아웃풋 비중이 크기 때문에 토큰 소모가 두드러진다고 분석했다. 실제로 어제 하루 3,500만 인풋 토큰 중 아웃풋이 140만으로 전체의 약 5%에 불과했지만, Linear 이슈 등록은 아웃풋 비중이 상대적으로 높다고 설명했다. 서진우 매니저는 아웃풋 토큰이 인풋 대비 약 3배 비싸다는 점을 확인했다.

한글 사용이 토큰 소모를 가중시킨다는 문제도 제기되었다. 박범균 소장은 같은 문장을 영어와 한글로 비교했을 때 한글이 약 4~5배 많은 토큰을 소모하며, 리즈닝(thinking) 토큰도 한글 입력 시 더 많이 사용된다는 테스트 결과를 공유했다. 한글을 사용하는 이상 이 문제는 구조적으로 해결이 어렵다고 결론지었다.

Linear 등록 시 Cloudflare가 특정 문자열을 공격으로 간주해 차단하면서 에이전트가 긴 내용을 재작성하는 경우도 낭비 요인으로 지목되었다. 박범균 소장은 이런 실패-재작성 루프가 하루에 여러 번 발생하기도 한다고 설명했다.

### Superpowers 플러그인 비용 문제

박범균 소장은 Superpowers 사용 시 플랜 및 디자인 문서 생성 과정에서 세션 사용량의 30% 이상이 단번에 소진되는 현상을 지적했다. 코드 작업 시작 전 Superpowers 브레인스토밍과 플랜 작성, Linear 등록까지 완료하면 세션 사용량의 50%가 이미 소진되는 구조라는 분석이었다. 문서 생성 시 내부 리즈닝에 시간이 오래 걸리는 점도 비용 증가 원인으로 추정했다.

이에 따라 Superpowers 제거를 고려 중이라고 밝혔다. 초기 방향 설정이나 개념 구체화 단계에서는 브레인스토밍 기능이 유용하지만, 디자인 문서 생성 등 나머지 기능은 최신 모델의 기본 성능으로도 충분히 대체 가능하다고 판단했다. 현재는 더 단순한 도구로 대체하는 방안을 검토 중이다.

## 5. Claude Code 운영 방식 및 플러그인 정리 (54:00~1:10:00)

### Andrej Karpathy 스킬 검토

박범균 소장이 GitHub에서 Andrej Karpathy가 공유한 "Claude Code Guidelines" 마크다운 스킬 파일을 검토 중이라고 소개했다. 약 50줄짜리 간결한 마크다운으로, 코딩 전 사고하기, 더 단순한 방법 탐색, 불필요한 코드 정리, 목표 기준 검증 루프 실행 등 4~6가지 원칙으로 구성되어 있다. 특히 "혼란이 발생하면 작업을 멈추고 불명확한 사항을 질문하라"는 원칙이 포함되어 있어 작업 중 불필요한 시행착오를 줄이는 데 효과적일 것으로 기대했다.

다만 이 스킬 적용 시 에이전트가 작업 중간에 자주 질문을 멈추고 물어볼 수 있다는 점이 우려 사항으로 제기되었다. 주정민 매니저는 실제 사용해본 결과 진행 중에도 여러 번 작업을 멈추고 5~6개씩 질문하는 경우가 있었다고 공유했다. 박범균 소장은 브레인스토밍 단계에서는 100개 질문도 문제없지만, 코딩 중간에 작업이 끊기는 것은 불편하다고 반응했다. 공통 스킬로 배포 전 프로젝트별 테스트가 필요하다는 결론이었다.

### 주석 정책 정리

김경승 팀장은 코드 수정 시마다 "Modified: 날짜, 내용" 형태의 주석이 자동으로 생성되어 SQL 파일 등에서 주석이 200줄 가까이 누적되는 문제를 제기했다. 이 주석들이 토큰 소모를 늘리고, AI의 코드 파악 정확도를 떨어뜨리는 원인이 되고 있다고 설명했다. 커밋 이력으로 충분히 추적 가능하므로 수정 이력성 인라인 주석은 제거하는 것이 맞다는 데 모두 동의했다. 다만 함수 파라미터나 독스(Docs) 형식의 기본 주석은 유지하는 방향으로 가이드라인을 보완하기로 했다. 기존 주석 제거 스크립트를 작성해 코드 양을 줄이는 작업도 진행하기로 했다.

### Claude.md 최적화 방향

박범균 소장은 현재 Claude.md 파일에 코드 작성 양식 등 새 코드 생성 시에만 필요한 내용이 포함되어 있다고 지적했다. 이를 별도 파일로 분리하고 Claude.md에는 "새 코드 작성 시 특정 파일 참조" 한 줄만 남기는 방식으로 컨텍스트 압축이 가능하다고 제안했다. 또한 특정 경로에 특이사항이 있을 경우 해당 경로에 클로드 `.md` 파일을 생성하면 Claude Code가 해당 경로 탐색 시 자동으로 읽어들인다는 점을 활용하자고 언급했다.

조직 단위 Claude 스킬 적용에 대해서는 모든 팀원의 모든 세션에 불필요한 내용이 올라가는 비용 문제가 있으므로 신중하게 판단해야 한다고 강조했다. 정말 모든 상황에서 반드시 지켜야 할 규칙만 조직 단위로 적용하는 것이 적절하다는 입장을 밝혔다.

## 6. 그래프 RAG 및 LLM Wiki 검토 (1:21:00~1:30:54)

김태규 매니저는 그래프 데이터베이스 기반 검색 정확도가 개선되어 반영 중이라고 공유했다. 수동 업데이트 중이나 추후 이벤트 기반 자동 반영 또는 일 1회 배치 처리로 전환할 예정이라고 밝혔다.

박범균 소장은 유튜브에서 접한 "LLM Wiki" 또는 "Graph RAG" 방식을 소개했다. 특정 경로에 문서들을 모아두면 LLM이 인덱싱하여 마크다운 파일들로 자동 구조화하고, 문서 간 참조 관계를 그래프로 연결하여 시각화하는 방식이다. 기존 벡터 RAG의 한계(임베딩 유사도 기반 검색, 임베딩 비용, 검색 정확도 문제)를 극복하는 접근법이라고 설명했다. 사내 매뉴얼 문서를 LLM이 검색하기 어려웠던 문제의 해결책이 될 수 있다고 기대하면서도, 현재는 직접 구현 의향은 없으며 테스트해볼 가치가 있다는 정도로 정리했다.
